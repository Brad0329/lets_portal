## 1. 문제 정의

### 1-1. 현상
- `backend/extractors/proposal.py` 의 `_segment_text()` 가 `Ⅰ./Ⅱ./Ⅲ.` 최상위 + `^[1-9]\s+[가-힣]` 중간 2개 정규식만 지원.
- 이 규칙에 안 맞으면 **섹션 0개 반환** → `target_sections = []` → LLM 호출 0회 → claim 0건.
- `status='completed'`, `cost=0`, `error=None` 로 "정상 완료"처럼 기록돼 조용한 실패가 됨.

### 1-2. 근본 원인
- **단일 샘플 검증의 과신**: Phase C-5 MVP 종료 시 경과원 ESG 제안서 1건만 돌려보고 "실증"으로 기록. 다른 형식은 한 번도 시도하지 않음.
- **설계-구현 gap**: 원 설계(`design_phase_c_extractor.md §5-1`)에 "폴백2: 글자수 기반 청크"가 명시됐으나 실구현 누락. 검토 단계에서 잡지 못함.
- **조용한 실패 허용**: 섹션 0개일 때 경고·에러 없이 completed 처리 → 운영자가 이상을 인지 못함.

### 1-3. 파급 범위
이 파서는 C-5(역량 자산)·C-5-B(자동 채우기) 양쪽의 **공통 엔트리 포인트**. 여기가 무너지면 위 기능 전부 실효성 없음.

## 3. 접근 전략 — 하이브리드 3단계 폴백

### 3-1. 전체 플로우

```
파일 업로드
  ↓
[1] 파일 → 마크다운 텍스트 추출  ← 현행 유지 (file_parser.py)
  ↓
[2] 섹션 분할 — 3단계 폴백
    ├─ 2-A. 목차 페이지 탐지 (Python)
    │     ↓ 성공 (우선순위 1)
    │     목차 항목을 본문에서 매칭 → 섹션 배열
    │
    ├─ 2-B. Haiku 섹션 분할 (목차 실패 시)
    │     ↓
    │     첫 2~3페이지만 Haiku에 투입 → 마커 패턴 JSON 반환
    │     → Python이 정규식 분할
    │
    └─ 2-C. Flat chunk 폴백 (모두 실패 시)
          ↓
          문단 경계 유지하며 20K자씩 강제 분할
          UI에 "섹션 분할 실패 — 강제 청크 사용" 경고
  ↓
[3] claim 추출 — 현행 유지 (Sonnet)
  ↓
결과 저장
```

### 3-2. 각 단계 설계

#### 2-A. 목차 페이지 탐지 (Python, 결정적)

제안서는 보통 1~5페이지 안에 목차가 있다. "목차 / 목 차 / CONTENTS / 차례 / Index" 라인을 앵커로 삼아 이후 수십 라인을 **목차 항목으로 수확**.

**알고리즘 (의사코드):**
```
lines = text.split("\n")[:200]  # 앞부분만
anchor = find_line(r"^\s*(목\s*차|CONTENTS|차례|Index)\s*$")
if not anchor: return None

toc_lines = []
for L in lines[anchor+1 : anchor+100]:
    if is_blank: break_count++ (3번 연속이면 목차 종료)
    if looks_like_toc_entry(L):   # "1. 제목 ... 3" 같은 패턴
        toc_lines.append(clean(L))

# 목차 → 본문 매칭
sections = []
for toc in toc_lines:
    idx = find_first_match(body, toc)  # toc 제목이 본문에 처음 나타나는 위치
    sections.append(Section(name=toc, start_line=idx, ...))
```

**목차 항목 인식 휴리스틱:**
- 점선·탭·공백 뒤 페이지 번호 패턴 (`1. 제안개요 ...... 3`)
- 마커(로마자, 숫자, 한글) + 제목 + 우측 여백 + 페이지 숫자

**실패 조건:**
- "목차"·"차례" 앵커 없음
- 목차는 있으나 본문 매칭 실패율 50% 초과

#### 2-B. Haiku 섹션 분할 (Haiku 4.5, 저가 LLM)

목차 탐지 실패 시 문서 **첫 2~3페이지(약 3K 토큰)** 만 Haiku에 투입. 과제: "이 문서의 섹션 제목 마커 패턴을 JSON으로 알려달라".

**프롬프트 개요:**
```
역할: 한국 제안서의 섹션 제목 패턴을 식별하는 전문가.
입력: 제안서 첫 2~3페이지 텍스트.
출력 JSON:
{
  "marker_examples": ["1. 제안개요", "2. 추진계획", ...],
  "regex_top": "^\\d+\\.\\s+[가-힣]",
  "regex_mid": "^\\d+\\.\\d+\\s+[가-힣]",
  "confidence": 0.92
}
```

**이후 처리:**
- Python이 반환된 정규식으로 전체 텍스트 재분할
- confidence < 0.7이면 2-C 폴백으로 이동

**비용:**
- 입력 3K·출력 200토큰 ≈ **$0.006 ≈ 8원/건** (Haiku 4.5 가격 기준)
- 프롬프트 캐싱 적용 시 더 저렴
- 모든 제안서에 적용해도 부담 없음. 다만 목차 탐지가 성공하는 경우(추정 60~70%)는 안 쓰므로 실제 평균 2~3원 수준.

#### 2-C. Flat chunk 폴백 (최후 방어선)

- 문단 경계 유지(`\n\n` 또는 빈 줄)하며 **20K자 단위 청크**
- 섹션 이름은 `"청크 1"`, `"청크 2"` 식
- UI에 **경고 배너** 표시 — "섹션 분할 실패. 강제 청크로 처리됨. 결과 품질이 떨어질 수 있음"

## 8. 구현 시 조심할 것

### 8-1. bidwatch 이식 원칙 유지
- 파서 리빌딩해도 `backend/extractors/`의 portable 원칙 준수
- 새 함수도 `database`·`routers`·`config` 등 lets_portal 특정 모듈 import 금지
- Haiku 클라이언트는 `backend/ai/` 아래 기존 구조에 추가 (이미 portable)

### 8-2. 기존 MVP 동작 보존
- 경과원 ESG 제안서(샘플 6번 `제안서(사본).hwp`)는 현재 파서로 정상 동작 중. 리빌딩 후에도 같은 품질 유지돼야 함.
- Regression 테스트에 경과원 샘플 포함 필수.

### 8-3. Haiku 품질 함정
- Haiku가 JSON 구조화 출력은 잘 하지만 창의적 요약은 품질 편차 큼
- 섹션 분할은 "식별" 과제라 OK. 이후 "추출"까지 확장할 때는 별도 실험 선행.

### 8-4. PPTX 처리 정책 — **미지원으로 확정 (2026-04-22, D-2 종료 시점)**

**결정:** PPTX/PPT 는 본 파서 지원 대상이 아니다. PDF 또는 DOCX 로 변환 후 업로드 안내.

**근거:**
- 샘플 10건 중 PPTX 1건(한국관광공사)뿐 — 전환 ROI 낮음
- 슬라이드 단위 구조가 섹션 개념과 근본적으로 다름. 슬라이드를 섹션으로 취급하면 "섹션당 1~2문장"이 되어 LLM 호출 단위로 과대 분할
- 사용자가 PDF 변환은 Office 에서 즉시 가능 (Export → PDF)

**구현:**
- `backend/utils/file_parser.py` 에서 `.pptx`/`.ppt` 확장자는 명시적 에러 메시지 반환
- UI 업로드 폼에 지원 포맷 안내 (별건 — 설정 페이지 프로필 업로드 UX 검토 시)

**롤백 조건:** PPTX 비중이 증가하면(예: 샘플 20건 중 5건+) `python-pptx` + 슬라이드 통합 파싱 재검토.

### 8-5. "조용한 실패" 방지

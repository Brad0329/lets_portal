### 실패 패턴 분류

**패턴 A — 마커가 "번호+점" 스타일 (조용한 실패 원인)**
- #2 혁신창업: `Ⅰ . 신청개요` (로마자와 점 사이 공백) + `1. 기관연혁` (서브섹션 점+공백+한글)
- #5 충남콘텐츠: `Ⅰ. 제안 개요` + `1. 사업 목적 및 범위`

현 `_MID_RE = ^([1-9])\s+([가-힣]...)$` 는 번호 직후 공백만 허용. `1.` 처럼 점이 낀 스타일을 전부 놓친다 → `current_mid` 가 한 번도 세팅되지 않아 `flush()` 내부 가드(`current_mid and buffer`)에서 막혀 섹션 0개로 끝난다. 현 파서가 `extractors.proposal` 에 들어간 당시의 단일 샘플(경과원 ESG)이 `1 사업의 필요성` (공백) 스타일이라 이 분기를 한 번도 검증하지 않은 것.

**패턴 B — 섹션명 로마자 누락 (품질 저하)**
- #7 한국저작권위: `.1 제안 배경 및 필요성`, `.2 사업목표`, … (앞에 `Ⅰ` 빠짐)
- 원인: HWP 렌더링 시 `Ⅰ` 로마자가 별도 단락으로 분리돼 `_TOP_RE` 에 걸리지 않고, 곧바로 `_MID_RE` 로 `1 제안…` 이 잡힘 → `current_top = None` 상태에서 `flush()` 가 `roman = ""` 로 이름을 붙여 `.1 제안…` 가 됨.

### 목차 앵커 탐지 전원 실패

10건 모두 내 진단 정규식으로 목차 발견 0건. 실제 덤프 텍스트를 열어보니 HWP→마크다운 변환 과정에서 **"목     차" 가 페이지 전체 내용(수십 개 목차 항목+페이지 번호)과 함께 한 줄에 밀려 들어간다**. 예 (혁신창업):

```
목     차 Ⅰ. 신청개요 1 1. 기관연혁 2 2. 협업기관 일반현황 5 … Ⅵ. 붙임자료 101
```

**D-2 시사점**
- 앵커 탐지는 `^\s*목\s*차\s*$` 같은 strict 매칭이 아니라 **라인 선두 "목 차" 키워드 포함** + **뒤에 목차 엔트리 토큰(`로마자 + 한글 + 숫자페이지`)이 K개 이상** 조합으로 가야 한다.
- 항목 추출도 "줄 단위 목차 라인"이 아니라 **한 줄 안에서 정규식으로 목차 항목 시퀀스 토큰화** 하는 방식이 필요. 설계 §3-2의 의사코드(`lines[anchor+1:anchor+100]`)는 이 HWP 출력 형태에서 동작 안 함.

### 결정사항

1. **"조용한 실패"를 허용하지 않는다 — `extract`가 섹션 0개로 끝나면 `ExtractionResult.warnings` 에 `{"type":"segment_empty"}` 기록 + `proposal_sources.status = 'warning'`**. 현행은 `completed/cost=0/error=None`로 UI가 정상 완료와 구분 못 함. D-4 이후 UI 배너와 연결.
2. **`_MID_RE` 는 D-2에서 즉시 수정하지 않는다 — 하이브리드 3단계 폴백의 2-A(목차 탐지)부터 구축.** 마커 정규식만 늘리면 또 다른 형식에서 같은 함정이 반복된다(현재 num_space_kr 대응 추가해도 num_dot+nested 등이 남음).
3. **Ground truth는 현 파서 결과 그대로가 아니라 "목차 기반 재구성"을 1차 후보로** — D-1 결과는 섹션이 잘못 잡히거나 비어 있어 ground truth 소스로 부적합. D-2에서 목차 탐지 프로토타입이 붙으면 그 출력을 1차 후보로 쓰고 사용자가 보정.
4. **진단 도구는 `tools/check_parser.py` 로 유지 — D-5 regression 스크립트는 별도**. check_parser는 "현황 보는 렌즈"(텍스트 덤프·마커 히스토그램), regression은 ground truth 대비 F1 산출로 역할이 다름.

### 실패한 접근 / 조심할 것

1. **진단 도구를 bash에서 돌리면 Windows cp949 콘솔 깨짐.** `print_file_report`가 한글을 찍는데 Git Bash 경유 stdout이 cp949로 디코딩돼 리포트가 모지바케. **해결책: JSON 덤프(`--dump`) 기본 사용**, 또는 PowerShell에서 `chcp 65001`. 앞으로 reporter 쓸 때는 파일 기반 결과가 일차 소스.
2. **14MB HWP 하나 파싱에 9.4분.** pyhwp가 tree walk할 때 표 셀이 많은 제안서에서 극단적으로 느려짐. D-5 regression을 매번 전체 10건으로 돌리면 ~14분. **대책: ground truth 생성 후 파싱 결과 텍스트를 캐시(`data/ground_truth/{name}.txt`)**, regression은 섹션 분할 단계부터 돌리는 구조로 설계.
3. **진단 정규식으로 잡은 `num_dot=28`(혁신창업)이 섹션 마커가 아니라 목차 라인 내 토큰의 부분 매칭**일 가능성. 한 줄에 `… 1. 기관연혁 2 2. 협업기관 …`로 흘러가 있어 ripgrep/정규식이 "문자열 시작 = 라인 시작"을 의미하는 `^`에서도 토큰 경계가 모호. 실제 마커 카운트와 목차 카운트 분리 집계가 필요하다면 2-A 구현 시 정제.

### 결정사항

1. **정규식 1개가 아닌 토큰 파서 채택** — 설계 §3-2 의사코드는 정규식 `(로마자·숫자)+제목+페이지` 하나로 탐지했다. D-1 실제 데이터는 `·`, `&`, `Ⅱ·Ⅰ`, 괄호 등 제목 내 특수문자가 다양해 정규식 greedy 제어가 불안정. **공백 분리 토큰을 상태 기계로 순회**해 마커 만나면 엔트리 시작 · 순수 숫자 토큰 만나면 페이지 확정. 견고성이 훨씬 높다.

2. **`body_start_line = anchor_line + 1`** — 최초 구현은 `anchor.end() + toc_span_chars(4000)` 위치 이후 첫 논-빈 라인으로 추정했는데, HWP 파서가 목차 전체를 한 줄에 밀어넣은 탓에 **본문 시작 한참 뒤**로 추정돼 앞단 섹션을 전부 miss(#8 경남 47%의 실제 원인). 앵커 라인 바로 다음부터 탐색하되 **커서 단조성**으로 역순 매칭 방지. 이 한 줄 수정이 전체 매칭률 72%→90%의 핵심.

3. **엔트리 품질 필터 3종** — 목차 영역 크기(4000자)를 넘어 본문이 섞여들면 False positive 발생. 방어책: (a) 제목 토큰 수 10개 초과 시 폐기, (b) 제목에 장식문자(▸○●◆…) 포함 시 폐기, (c) 페이지 번호 2회 연속 5+ 역행 시 이후 엔트리 수확 중단. #7 원광대의 `Ⅴ.4 결과 보고 ▸ 운영 결과물 도출…` 같은 오인 케이스를 잡아냄.

4. **3단계 매칭 폴백** — strict(마커+제목 정확) → relaxed(제목만 유사+마커 prefix 확인) → approx(제목 앞 3토큰 + 마커 prefix). D-1 관찰 결과 같은 섹션 제목이 `1. 제목`·`1 제목`·`1) 제목` 여러 형태로 본문에 나타나 엄격 매칭만으로는 부족.

5. **bidwatch 이식 원칙 준수** — `segmenter/` 패키지 전체가 `database`·`routers`·`config` 등 lets_portal 특정 모듈 import 안 함. `extractors.base` 의 `Section` 도 직접 참조하지 않고 자체 `MatchedSection` 데이터클래스 사용(이식 시 충돌 감소).

6. **현 `proposal.py` 의 `_segment_text` 교체는 세션 3 이후로 연기** — D-3(Haiku 폴백) 결합해서 한 번에 교체. 지금은 `segmenter` 가 독립 도구로만 존재.

### 실패한 접근 / 조심할 것

1. **정규식 하나로 greedy 제어 시도 → 실패** — `(?P<title>.+?)\s+(?P<page>\d{1,3})` 는 제목에 `·` 들어가면 예상과 다르게 매칭. 실제 샘플은 `내 · 외부 인력의 전문성 29` 같이 점이 있어 `.+?`가 다음 마커 직전에서 끊기지 않고 과도/과소 매칭. 토큰 파서로 전환 후 해결.

2. **`body_start_line` 과도 보수 추정** — 목차가 짧고 본문 시작이 바로 뒤따르는 HWP 레이아웃에서 `toc_span_chars=4000` 이후를 본문 시작으로 잡으면 앞 섹션 전부 놓침. 설계 §3-2 가 "목차 영역"을 글자 수로 가정한 것이 HWP 파싱 결과와 안 맞음. **단조성 커서만 있으면 start_line 은 보수적일 필요 없다**.

3. **콘솔 한글 깨짐은 또 발생** — Git Bash/cp949 환경. `run_toc.py` 의 print_report 에서 status 'parse_fail' 이유가 깨짐. 대책: 파일별 결과는 항상 `--dump` JSON 으로 확인. check_parser 세션 1의 잠재 과제와 동일.
